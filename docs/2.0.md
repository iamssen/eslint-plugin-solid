# Solid 2.0 대응 검토

`docs/Solid2-beta-Migration-Guide.md`의 베타 API를 기준으로 `cases/`와 `src/rules/`를 검토한 결과이다. 이 문서는 구현 계획이며, 아직 규칙이나 fixture를 Solid 2.0으로 변경한 것은 아니다. Solid 2.0은 베타이므로 실제 반영 전에는 사용하려는 Solid 2.0 버전의 export/type을 다시 확인해야 한다.

## 결론

현재 플러그인의 권장 설정과 통합 fixture는 Solid 1.x를 기준으로 한다. 특히 `imports`, `no-destructure`, `reactivity`, `no-unknown-namespaces`, `event-handlers`는 2.0에서 잘못된 API를 허용하거나 자동 수정 결과로 만들 수 있다. `cases/valid`도 다수의 1.x 코드를 정상 코드로 고정하므로, 규칙을 먼저 바꾸면 통합 테스트가 의도적으로 실패한다.

따라서 다음 순서가 안전하다.

1. `imports`와 JSX/TypeScript 기준을 2.0으로 전환하고, fixture의 import 경로를 일괄 이전한다.
2. 제거된 JSX 문법을 진단하도록 규칙을 갱신한다.
3. 자동 수정이 1.x helper를 만들지 않도록 `no-destructure`와 `reactivity`를 전환한다.
4. effect, async, store, control-flow의 의미 변경을 전용 호환성 규칙과 fixture로 보강한다.

기존 1.x 사용자도 지원해야 한다면 이를 기존 `recommended`에 바로 넣기보다 `solid2`/`recommended-2` 같은 별도 flat config로 시작하는 편이 호환성 파괴를 줄인다. 2.0 전용이 이 패키지의 기본 목표라면 기존 `recommended`를 바꾸되 major release로 취급해야 한다.

## 우선순위 1: import와 JSX 타입 소유권

### `imports` 규칙

`src/rules/imports/rule.ts`는 source를 `solid-js`, `solid-js/web`, `solid-js/store` 세 경로로 한정하고, `render`를 `solid-js/web`, store API를 `solid-js/store`로 자동 수정한다. 이는 2.0에서 모두 잘못된 결과다.

| 대상 | 현재 규칙/fixture | 2.0 기준 |
| --- | --- | --- |
| web renderer | `solid-js/web` | `@solidjs/web` |
| store API | `solid-js/store` | `solid-js` |
| `createStore`, `reconcile` 등 | store source map | core source map |
| JSX 및 `ComponentProps` 타입 | `solid-js` type map | 웹 JSX는 `@solidjs/web` |
| hyperscript/HTML/universal | 추적하지 않음 | `@solidjs/h`, `@solidjs/html`, `@solidjs/universal` |

필요한 변경은 다음과 같다.

- `Source`와 source 정규식을 `@solidjs/*` 경로까지 포함하도록 다시 설계한다. 단순 문자열 치환이 아니라 기존 import 병합/삽입 fixer가 새 source에서도 동작하도록 테스트한다.
- 1.x 전용 export(`createResource`, `batch`, `Index`, `Suspense` 등)는 올바른 source로 옮기는 대신 별도 “제거된 API” 진단으로 보고한다. 존재하지 않는 export를 `solid-js`로 자동 수정하면 오류를 숨기게 된다.
- `JSX`, `ComponentProps` 같은 타입의 목적지를 구분한다. renderer 중립 자식 타입은 `solid-js`의 `Element`를 사용하고, 웹 JSX 타입은 `@solidjs/web`으로 유도한다.
- 규칙 문서 URL은 아직 원본 프로젝트 경로를 가리키므로, 2.0 작업과 함께 이 하드 포크의 GitHub 경로로 정정한다.

`src/rules/imports/rule.test.ts`의 valid/invalid/output은 전부 1.x source를 전제한다. 새 경로, type-only import, alias, 한 선언 안에서 core/web specifier가 분리되는 경우를 추가해야 한다.

### TypeScript 설정과 fixture

저장소의 `tsconfig.json`은 `cases/valid`을 type-check 대상에서 제외하며 JSX 옵션도 없다. 통합 fixture는 ESLint parser만 사용하므로, `jsxImportSource` 이전을 검증하지 못한다.

- Solid 2 웹 fixture용 tsconfig(또는 별도 type test)를 만들고 `jsx: "preserve"`, `jsxImportSource: "@solidjs/web"`을 검증한다.
- `solid-js/jsx-runtime`과 `solid-js/jsx-dev-runtime` import를 금지/이전하는 테스트를 둔다.
- `cases/valid`의 `solid-js/web` import는 대부분 `@solidjs/web`으로 이전해야 한다. store import도 `solid-js`로 합쳐야 한다. 대표적으로 `stores/create-store`, `stores/mutation`, `stores/immutable-stores/useRedux`, `examples/formvalidation-*`, `examples/scoreboard`, `examples/simple-todos`가 해당한다.

## 우선순위 1: 제거된 JSX namespace와 class API

### namespace 규칙군

Solid 2.0 공개 JSX 모델에서는 `use:`, `attr:`, `bool:`, `on:`, `oncapture:`가 제거되며, `class:`/`style:`도 특별한 문법이 아니다. 반면 현재 `no-unknown-namespaces`는 이들을 모두 정상 namespace로 허용한다. `event-handlers`는 `attr:on...`을 수정 제안으로 만들고, `no-array-handlers`, `jsx-no-undef`, `reactivity`도 `on:`/`use:`를 유효한 특수 문법 또는 reactive scope로 가정한다.

| 규칙 | 필요한 2.0 변경 |
| --- | --- |
| `no-unknown-namespaces` | 제거된 namespace를 허용 목록에서 빼고, 2.0 전용 오류 메시지와 대체 방향을 제공한다. `prop:`은 native/custom element의 writable DOM property 설정으로 허용하며, SVG/XML namespace도 유지한다. |
| `event-handlers` | `attr:${name}` 제안 및 fixer를 제거한다. 일반 이벤트는 `onClick`을, native option/capture가 필요하면 `ref`에서 `addEventListener`를 쓰도록 메시지를 바꾼다. |
| `no-array-handlers` | `on:` namespace 전용 사례를 제거한다. array handler는 2.0에서 지원되므로 recommended에서 끈 opt-in 스타일 정책으로만 유지한다. |
| `jsx-no-undef` | `use:X` directive 식별자 검사 경로와 관련 테스트를 제거하거나 `ref={X(...)}` factory 사용 패턴에 맞춘 별도 검사를 설계한다. |
| `reactivity` | `use:`가 tracked scope라는 분기와 관련 테스트를 삭제하고, `ref` directive factory의 설정 단계/적용 단계를 오인하지 않도록 별도 모델을 만든다. |

`cases/valid/bindings/directives/main.tsx`, `cases/valid/reactivity/on/main.tsx`, `cases/valid/examples/formvalidation-main.tsx`는 실제 `use:` 또는 `on:` fixture다. 각각 `ref={directive(options)}` 및 camelCase handler/ref listener 패턴으로 바꾸고, directive의 cleanup과 복수 ref 배열도 사례에 포함해야 한다.

### `class:` / `style:`

`class:`와 `style:`은 Solid 2.0의 특별한 JSX 문법이 아니다. `no-unknown-namespaces`의 관련 valid test와 문서를 2.0 기준 invalid test로 바꾼다. JSX의 조건부 class는 `class`의 객체·배열 값으로 작성하며, DOM API의 `element.classList` 사용은 JSX prop과 다르므로 그대로 둔다.

## 우선순위 1: 자동 수정과 reactivity 모델

### `no-destructure`

`no-destructure`는 rest prop을 `const props = _props; const rest = omit(props, ...keys);`로 자동 수정한다. 이름 있는 prop이 없으면 `omit(props)`를 사용한다. 2.0의 `omit`은 1.x `splitProps`처럼 tuple을 반환하지 않는다.

parameter default는 report-only다. `merge`의 `undefined`는 덮어쓰기 값이므로 `merge(defaults, _props)`를 삽입하면 JavaScript parameter default의 의미를 보존하지 못한다. default + rest와 computed key도 같은 이유로 자동 수정하지 않는다.

`reactivity.ts`는 `merge`와 `omit`의 단일 object 반환값을 reactive props로 추적한다. 새 store/helper(`snapshot`, `storePath`, `createProjection`, 함수형 `createSignal`/`createStore`)의 반환 형태도 필요한 범위에서 추가해야 한다.

### 효과, 배치, 쓰기 진단

가이드의 핵심은 setter 뒤 값이 즉시 읽히지 않고 microtask flush 뒤 갱신된다는 점, 그리고 계산/추적 scope 안의 reactive write가 개발 환경에서 오류가 된다는 점이다. `reactivity`는 Solid 2 `createEffect`의 compute callback 안에서 signal/store setter를 호출하는 문제를 진단하고, apply callback과 `{ effect, error }` bundle callback은 호출 시점 scope로 처리한다.

2.0 전용 분석을 추가해야 한다.

- `createMemo` 계산 함수 안의 signal/store setter 호출도 보고할지 별도로 결정한다. `createEffect`의 두 인수/`EffectBundle` 형태는 구분한다.
- 컴포넌트 최상위의 `props.x` 읽기와 parameter destructuring은 2.0 dev warning 대상이다. 기존 `no-destructure`/`reactivity`는 상당 부분을 다루지만, `const value = props.x` 외의 alias·초기화·`untrack` 의도 표기를 회귀 사례로 보강한다.
- `createEffect(fn, initialValue)`, `createMemo(fn, initialValue)`, `on(...)`, `createComputed`, `batch`, `onMount` 등의 1.x 호출을 import alias까지 추적해 보고한다. `no-react-deps`는 명확한 primitive initial value와 dependency array를 이미 진단하며, 나머지 1.x 호출은 전용 migration rule에서 다룬다.
- setter 직후 accessor를 읽는 동기 가정은 AST만으로 완전한 자동 수정이 어렵다. `flush()` 삽입은 시스템 flush와 의미 변경 위험이 있으므로 자동 수정하지 말고, 감지 가능한 동일 block 패턴에 경고/문서 제안만 제공한다.

## 우선순위 2: 제거 API 호환성 규칙

`imports`는 “올바른 경로”만 판단하는 규칙이므로, 제거 API는 전용 rule(예: `no-solid-1-apis`)로 분리하는 것이 좋다. named import, alias, namespace import와 JSX tag를 모두 추적하고, 대체가 한 가지로 확정되지 않는 항목은 자동 수정하지 않는다.

| 1.x API/문법 | 2.0 방향 | 현재 영향 범위 |
| --- | --- | --- |
| `createResource`, `Suspense`, `SuspenseList`, `useTransition` | async `createMemo`/`createStore`, `Loading`, `Reveal`, `isPending`/`action`/optimistic API | `cases/valid/async/**`, `examples/async-resource.tsx`, `examples/suspense-transitions-*` |
| `ErrorBoundary` | `Errored` | `control-flow/error-boundary/main.tsx` |
| `Index`, `indexArray` | `<For keyed={false}>`, `mapArray` | `control-flow/index/main.tsx`, `prefer-for` 문서/메시지, `reactivity`의 keyed=false accessor 처리 |
| `onMount` | `onSettled` | lifecycle fixture, `reactivity`의 called-function scope |
| `onMount` | `onSettled` | lifecycle 및 ref/forwarding-ref fixture, `reactivity` tracked-scope 목록 |
| `mergeProps`, `splitProps` | `merge`, `omit` | props fixture, `no-destructure`, `reactivity`, `no-proxy-apis` |
| `produce`, `createMutable`, `modifyMutable`, `unwrap` | draft setter, `createStore`, `snapshot` | store mutation fixture, import map/reactivity 추적 |
| `batch` | 기본 microtask batch, 필요 시 `flush()` | `reactivity/batching-updates/main.tsx`, reactivity sync-callback 목록 |
| `onError`, `catchError`, `from`, `observable`, `createDeferred` 등 | `Errored`/effect error, async iterator/effect, 외부 adapter | `imports`의 legacy export map 및 새 rule test |
| `Context.Provider` | `<Context value={...}>` | `cases/valid/stores/context/counter.tsx` |

`createResource`와 transition은 단순 rename이 아니므로 fixture를 기계적으로 바꾸지 않는다. 초기 fallback(`Loading`), 입력 변경 중 표시(`isPending`), 단독 `refresh()`의 조용한 재검증, optimistic update를 각각 별도 사례로 재작성해야 한다. `SuspenseList`의 `revealOrder`/`tail`도 `Reveal`의 `order`/`collapsed`와 의미가 완전히 같지 않으므로 manual migration 대상으로 둔다.

`For`는 기본/`keyed={true}`일 때 item 값과 index accessor를, `keyed={false}`일 때 item accessor와 숫자 index를 전달한다. 따라서 `Index` fixture와 `reactivity`의 callback parameter 추적은 `keyed`의 literal 값까지 고려하도록 고쳐야 한다. `prefer-for`는 `<Index>` 언급을 제거하고, keyed mode가 callback 형태를 바꾸므로 한 item 인자 map 외에는 자동 수정하지 않는다.

## 통합 fixture 전환 계획

`cases/index.test.ts`는 `valid` 아래 결과가 경고 0개인 것을 요구한다. 다음 그룹을 먼저 2.0 fixture로 바꾸고, 변경 전에는 각각 old-API invalid fixture를 추가해 진단을 고정한다.

1. 모든 renderer/store import: `solid-js/web` → `@solidjs/web`, `solid-js/store`의 Solid store export → `solid-js`.
2. `bindings/classlist`, `bindings/directives`, `bindings/events`, `reactivity/on`: 새 JSX attribute/event/ref 모델.
3. `props/default-props`, `props/splitting-props`, `stores/mutation`, `stores/context`: `merge`/`omit`, draft setter, context provider 모델.
4. `control-flow/index`, `control-flow/error-boundary`, `async/suspense*`, `async/resources`, `async/transitions`, `lifecycles/onMount`, `reactivity/batching-updates`: control flow, async, lifecycle, batch 의미 전환.

각 rule의 RuleTester에는 최소한 다음을 넣는다.

- 2.0 valid 코드와 1.x invalid 코드가 import alias와 type-only import에서도 동일하게 판정되는지
- fixer가 `@solidjs/web`, `merge`, `omit`, `class` 배열/객체 같은 2.0 문법만 출력하는지
- 자동 수정할 수 없는 async/effect/transition migration은 fixer가 없고, 진단 메시지에 안전한 대체 방향이 담기는지
- `For keyed={false}`의 item/index callback 형식, ref directive factory와 cleanup, effect apply 단계의 write 허용 여부

## 설정과 배포 전 확인

- `recommended`가 1.x 호환을 계속 제공할지, 2.0 전용으로 바꿀지 공개 API 결정을 먼저 한다. 전자라면 `solid2` config를 추가하고 두 config의 fixture를 분리한다.
- 새 규칙의 metadata, schema, 메시지, 한국어 rule 문서를 함께 추가한다. README는 영어로 2.0 config 사용법과 renderer type 설정을 갱신한다.
- 규칙 문서의 원본 `solidjs-community/eslint-plugin-solid` URL을 이 저장소의 실제 GitHub 경로로 정정한다.
- 구현 후 `npm run lint`, `npm run type-check`, `npm test`, `npm run build`를 실행한다. parser 차이와 JSX namespace/type 동작까지 바뀌므로 최종적으로 `npm run test:all`도 필요하다.
