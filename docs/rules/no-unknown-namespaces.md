# no-unknown-namespaces

## 1. 규칙이 존재하는 이유 (Solid.js 1.0 기준)

Solid.js의 JSX 컴파일러는 요소의 속성에 특별한 접두사(네임스페이스, Namespace)를 붙여 브라우저 DOM API와 밀접하게 상호작용하는 강력한 기능들을 제공합니다. 

Solid.js에서 공식적으로 지원하는 네임스페이스는 다음과 같습니다:
- **`on:`** (이벤트 리스너 부착)
- **`oncapture:`** (캡처링 단계 이벤트 리스너)
- **`use:`** (커스텀 디렉티브 사용)
- **`prop:`** (요소의 프로퍼티 강제 지정)
- **`attr:`** (요소의 HTML 어트리뷰트 강제 지정)
- **`style:`** (개별 CSS 스타일 속성 지정)
- **`class:`** (개별 CSS 클래스 토글링)

하지만 개발자가 오타를 내거나 잘못된 지식으로 정의되지 않은 네임스페이스(예: `bind:`, `model:`, `unknown:`)를 사용할 경우, Solid.js 컴파일러는 이를 처리하지 못해 오류를 내뿜거나 완전히 무시해버려 심각한 버그로 이어질 수 있습니다.

이 규칙은 앞서 나열된 공식 네임스페이스 목록 이외의 알 수 없는 네임스페이스가 사용되었을 때 오류를 발생시켜, 렌더링 런타임 전에 문제를 조기에 차단하기 위해 존재합니다.

## 2. Solid.js 2.0에서의 변경 여부

새로운 네임스페이스가 향후 컴파일러 기능 추가와 함께 생겨날 가능성은 열려있지만, 알려지지 않은 임의의 네임스페이스를 차단한다는 기본 원칙은 Solid.js 2.0에서도 여전히 유지될 것입니다. 만약 2.0에서 공식 네임스페이스가 추가된다면, 규칙의 허용 목록(Allowlist)이 갱신되는 형태로 대응될 것입니다.

## 3. 규칙의 이해를 돕기 위한 추가 설명 및 예제

### ❌ 잘못된 사용 예 (알 수 없는 네임스페이스 사용)

아래 코드들은 Solid.js에 존재하지 않는 네임스페이스를 사용하여 개발자의 의도대로 작동하지 않는 예시들입니다.

```jsx
// ❌ Svelte 등 다른 프레임워크의 네임스페이스를 헷갈려 사용한 경우
let el1 = <input bind:value={mySignal} />;

// ❌ 오타로 인해 잘못된 네임스페이스 사용
let el2 = <div css:display="none" />;

// ❌ 존재하지 않는 임의의 네임스페이스 사용
let el3 = <div unknown:prop="value" />;
```

### 🟢 올바른 사용 예 (공식 네임스페이스 사용)

항상 Solid.js에서 지원하는 유효한 네임스페이스만 사용해야 합니다.

```jsx
// 🟢 on: 네임스페이스 (이벤트 부착)
let el1 = <button on:click={() => console.log('클릭')} />;

// 🟢 class: 네임스페이스 (클래스 토글링)
let el2 = <div class:is-active={isActive()} />;

// 🟢 style: 네임스페이스 (개별 스타일 적용)
let el3 = <div style:background-color="red" />;

// 🟢 use: 네임스페이스 (커스텀 디렉티브)
let el4 = <div use:myDirective={options} />;
```

### 시각적 요약

Solid.js 컴파일러는 `namespace:name` 구문을 만나면 앞의 `namespace` 부분을 파싱하여 특수한 처리를 수행합니다. 이 기능은 약속된 마법 단어(Keyword)에만 반응하도록 짜여 있으므로, 허용되지 않은 네임스페이스를 쓰면 마치 없는 주문을 외우는 것과 같이 아무 동작도 발생하지 않습니다.
